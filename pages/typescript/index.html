TypeScript ​

TypeScript中文网

个人理解TypeScript的宗旨（或者说我们为什么要用TypeScript）：在编译阶段完美通过所有的静态类型检查，能避免大部分的运行时错误的产生，提升代码的健壮性。类型描述的越合理、越准确代码就越健壮。但不是每个类型都要缩小到最小，代码健壮性的前提下充分考虑拓展性以及复杂度，才能更合理、更准确。（过犹不及）

匿名函数 ​

// 函数的上下文类型推断--参数 s 被推断为字符串类型

  console.log(s.toUpperCase());
});

// 也适用于箭头函数
names.forEach((s) => {
  console.log(s.toUpperCase());

类型别名和接口的区别

主要区别：接口可以被扩展，而类型别名不行

  • 类型别名不得参与在声明合并中，但接口可以。
typescript
// An interface can be re-opened
// and new values added:

interface Mammal {
  genus: string;
}

interface Mammal {
  breed?: string;
}

const animal: Mammal = {
  genus: "1234",
  // Fails because breed has to be a string
  breed: 1, 
};

type Reptile = {

  genus: string;
};

// You cannot add new variables in the same way
type Reptile = {

  breed?: string;
};

null 和 undefined ​

JavaScript 有两个基础值用于表示值不存在或未初始化的值：null 和 undefined。

TypeScript 有两个对应的同名类型。这些类型的行为取决于你是否启用了 strictNullChecks 选项。

关闭 strictNullChecks ​

可能是 null 或 undefined 的值仍然可以正常访问，并且值 null 和 undefined 可以分配给任何类型的属性。这类似于没有空检查的语言（例如 C#、Java）的行为方式。

开启 strictNullChecks ​

当值为 null 或 undefined 时，你需要在对该值使用方法或属性之前测试这些值。就像在使用可选属性之前检查 undefined 一样，我们可以使用缩小来检查可能是 null 的值

typescript
function doSomething(x: string | null) {
  if (x === null) {
    // do nothing
  } else {
    console.log("Hello, " + x.toUpperCase());
  }
}

非空断言运算符 ​

TypeScript 还具有一种特殊的语法，可以在不进行任何显式检查的情况下从类型中删除 null 和 undefined。在任何表达式之后写 ! 实际上是一个类型断言，该值不是 null 或 undefined

typescript
function liveDangerously(x?: number | null) {
  // No error
  console.log(x!.toFixed());
}
// 就像其他类型断言一样，这不会改变代码的运行时行为，所以当你知道值不能是 null 或 undefined 时，只使用 ! 很重要。

extends 和 & 的区别

在 TypeScript 中，extends 和 & 都可以用来组合类型，但它们的用途和行为是不同的。

extends 用于泛型约束，它表示一个类型必须是另一个类型的子类型。例如：

typescript
function example<T extends { a: number }>(arg: T): T {
  return arg;
}
1
2
3

在这个例子中，T 必须是一个包含 a 属性且属性类型为 number 的类型的子类型。

而 & 是交叉类型（Intersection Types），它将多个类型合并为一个类型。例如：

typescript
type Combined = { a: number } & { b: string };
1

在这个例子中，Combined 类型同时具有 { a: number } 和 { b: string } 的所有属性。

如何选择使用 extends 还是 & 取决于你的需求。如果你需要表示一个类型是另一个类型的子类型，那么应该使用 extends。如果你需要将多个类型的属性合并到一个类型中，那么应该使用 &。

/**
 * TypeScript 提供了三种成员可见性修饰符：public、private 和 protected。默认情况下，所有成员都是 public 的。
 * public 成员可以在任何地方访问。
 * private 成员只能在声明它们的类中访问。
 * protected 成员在声明它们的类和任何派生类中都可以访问。
 *
 * private 和 protected 仅在类型检查期间强制执行，不影响运行时
 */

class Animal {
  public name: string;
  private age: number;
  protected weight: number;

  constructor(name: string, age: number, weight: number) {
    this.name = name;
    this.age = age;
    this.weight = weight;
  }

  public sameAs(other: Animal) {
    /**
     * 允许跨实例访问private成员
     */
    return other.age === this.age;
  }
}

const animal = new Animal("dog", 1, 10);
console.log(animal.name);
console.log(animal.age); // 属性“age”为私有属性，只能在类“Animal”中访问。ts(2341)

class Dog extends Animal {
  /**
   * 派生类遵循基类契约，但可以选择公开更多功能的基类子类型，protected => public
   * 但不能公开private成员，因为private成员只能在声明它们的类中访问
   */
  weight = 10;
}

class Base {
  protected x: number = 1;
}
class Derived1 extends Base {
  protected x: number = 5;
}
class Derived2 extends Base {
  f1(other: Derived2) {
    other.x = 10;
  }
  f2(other: Derived1) {
    /**
     * 不能跨层级访问protected成员
     * Derived1拥有受保护成员x，继承自Base，Base也有受保护成员，只能在Derived1内或它的派生类里访问受保护成员
     */
    // Error: Property 'x' is protected and only accessible through an instance of class 'Derived2'.ts(2445)
    other.x = 10;
  }

装饰器组合 ​

在 TypeScript 中，装饰器组合是指将多个装饰器应用到同一个声明上。装饰器是一种特殊类型的声明，可以附加到类声明，方法，访问器，属性或参数上。装饰器使用 @expression 形式，其中 expression 必须求值为一个函数，该函数在运行时被调用，被装饰的声明信息作为参数。

装饰器评估 ​

装饰器评估是指装饰器的执行顺序和方式。装饰器是在编译阶段就确定的，但是它们的执行是在运行时进行的。

装饰器的评估顺序遵循以下规则：

  1. 参数装饰器，然后是方法，访问器，或属性装饰器应用到每个实例成员。
  2. 参数装饰器，然后是方法，访问器，或属性装饰器应用到每个静态成员。
  3. 参数装饰器应用到构造函数。
  4. 类装饰器应用到类。

类装饰器 ​

类装饰器在类声明之前声明。类装饰器应用于类构造函数，可以用来监视，修改或替换类定义。类装饰器不能用于声明文件中的声明或任何其他环境（例如，在声明文件中或在 declare class）中声明的任何其他类型的声明。

参数装饰器 ​

参数装饰器声明在一个参数声明之前（紧靠着参数声明）。参数装饰器应用于类构造函数或方法声明。参数装饰器不能用在声明文件中（.d.ts），或者任何外部上下文（比如 declare class）里。

参数装饰器表达式会在运行时当作函数被调用，传入下列3个参数：

  • 对于静态成员来说是类的构造函数，对于实例成员是类的原型对象。
  • 成员的名字。
  • 参数在函数参数列表中的索引。
typescript
import "reflect-metadata";
const requiredMetadataKey = Symbol("required");

class BugReport {
  type = "report";
  title: string;

  constructor(t: string) {
    this.title = t;
  }

  @validate
  print(@required verbose: boolean) {
    if (verbose) {
      return `type: ${this.type}\ntitle: ${this.title}`;
    } else {
      return this.title;
    }
  }
}

function required(
  target: Object,
  propertyKey: string | symbol,
  parameterIndex: number
) {
  let existingRequiredParameters: number[] =
    Reflect.getOwnMetadata(requiredMetadataKey, target, propertyKey) || [];
  existingRequiredParameters.push(parameterIndex);
  Reflect.defineMetadata(
    requiredMetadataKey,
    existingRequiredParameters,
    target,
    propertyKey
  );
}

function validate(
  target: any,
  propertyName: string,
  descriptor: TypedPropertyDescriptor<Function>
) {
  let method = descriptor.value!;

  descriptor.value = function () {
    let requiredParameters: number[] = Reflect.getOwnMetadata(
      requiredMetadataKey,
      target,
      propertyName
    );
    if (requiredParameters) {
      for (let parameterIndex of requiredParameters) {
        if (
          parameterIndex >= arguments.length ||
          arguments[parameterIndex] === undefined
        ) {
          throw new Error("Missing required argument.");
        }
      }
    }
    return method.apply(this, arguments);
  };
}

const bugReport = new BugReport("bug");
console.log(bugReport.print(true));

JSX ​

JSX 是一种嵌入式的类似 XML 的语法。它可以被转换成合法的 JavaScript，尽管转换的语义是依据不同的实现而定的。 JSX 被 React 用来描述用户界面，但也可以用于其他场景。TypeScript 支持内嵌，类型检查以及将 JSX 直接编译为 JavaScript。

基本用法 ​

为了使用jsx，必须完成以下两步

  • 使用.tsx作为文件扩展名
  • 启用jsx选项。tsconfig.json选项

TypeScript有三种jsx模式: preserve、react和react-native。这些模式只在代码生成阶段起作用 - 类型检查并不受影响。

  • preserve：在输出中保留JSX，不进行转换。例如，输入中的<div />输出为<div />。
  • react：JSX将被转换为使用React.createElement的形式。 例如，输入中的<div />输出为React.createElement("div")。
  • react-native：类似于preserve，但添加了一些适用于React Native的特殊情况。 例如，输入中的<div />输出为<div />。

as ​

常见的类型断言写法const foo = <foo>bar，但这种写法在.tsx中是不被允许的。在.tsx中使用const foo = bar as foo来代替。

更多.tsx类型检查和jsx的使用方法在React项目中进行体验和学习吧。用法和jsx用法无异，只是在.tsx中增加了相应的类型检查。

命名空间和模块 ​

在TypeScript中，命名空间（Namespace）和模块（Module）是用于组织和管理代码的两种不同的概念。

命名空间（Namespace）：

  • 命名空间是一种将相关的代码块封装在一起的方式，以避免全局命名冲突。
  • 命名空间使用namespace关键字定义，并使用点号.进行嵌套。
  • 命名空间可以包含变量、函数、类等代码实体。
  • 命名空间中的代码可以通过export关键字导出，以便在其他文件中使用。
  • 在使用命名空间中的代码时，需要使用namespaceName.entityName的方式进行访问。
  • 命名空间可以跨多个文件进行扩展，通过/// 指令引用其他文件。

模块（Module）：

  • 模块是一种将相关的代码封装在一起的方式，以便在不同的文件或项目中进行复用。
  • 模块使用import和export关键字进行导入和导出。
  • 模块可以包含变量、函数、类等代码实体。
  • 模块中的代码通过export关键字导出，以便在其他模块中使用。
  • 在使用模块中的代码时，需要使用import语句导入相应的模块，并使用moduleName.entityName的方式进行访问。
  • 模块可以通过export default关键字指定默认导出，使得在导入时可以省略实体名称。

相比而言，模块是更加现代化和推荐的组织代码的方式。在较新的TypeScript版本中，推荐使用模块来代替命名空间，因为模块提供了更好的代码隔离和可组合性。命名空间仍然可以在一些旧有的代码中使用，但在新项目中，应优先选择模块来组织代码。

unique symbol ​

unique symbol 是一种新的原始数据类型，它表示具有唯一值的类型。它的主要用途是用于对象属性键和常量，以及一些需要唯一标识符的场景。

typescript
const key1: unique symbol = Symbol("key");
const key2: unique symbol = Symbol("key");

  [key1]: "value1",
  [key2]: "value2",

console.log(key1); // Symbol(key)
console.log(key2); // Symbol(key)
// 由于unique symbol是唯一的，所以key1和key2是不相等的

console.log(obj[key2]); // value2

最佳通用类型 ​

当需要从几个表达式中推断类型时候，会使用这些表达式的类型来推断出一个最合适的通用类型。例如，

typescript
let x = [0, 1, null];
1

为了推断 x 的类型，我们必须考虑所有元素的类型。这里有两种选择：number和null。 计算通用类型算法会考虑所有的候选类型，并给出一个兼容所有候选类型的类型。

由于最终的通用类型取自候选类型，有些时候候选类型共享一个公共结构，但是却没有一个类型能做为所有候选类型的超级类型。例如：

typescript
let zoo = [new Rhino(), new Elephant(), new Snake()];
1

这里，我们想让 zoo 被推断为 Animal[] 类型，但是这个数组里没有对象是 Animal 类型的，因此不能推断出这个结果。 为了更正，当候选类型不能使用的时候我们需要明确的指出类型：

typescript
let zoo: Animal[] = [new Rhino(), new Elephant(), new Snake()];
