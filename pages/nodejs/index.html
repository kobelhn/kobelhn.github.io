是什么 ​

Node.js 是一个基于 Chrome V8 引擎的 JavaScript 运行环境。

Node.js 是一个开源、跨平台的 JavaScript 运行时环境。它是几乎适用于任何类型项目的流行工具！ Node.js 在浏览器外部运行 V8 JavaScript 引擎（Google Chrome 的核心）。这使得 Node.js 具有非常高的性能。 Node.js 应用程序在单个进程中运行，无需为每个请求创建新线程。 Node.js 在其标准库中提供了一组异步 I/O 原语，可防止 JavaScript 代码阻塞，并且通常 Node.js 中的库是使用非阻塞范例编写的，这使得阻塞行为成为例外而不是常态。

为什么用Node.js ​

Node.js 是一个跨平台运行时，非常适合各种用例。其庞大的社区使其易于上手。它使用V8引擎来编译JavaScript并以闪电般的速度运行。 Node.js 应用程序具有很强的可扩展性和可维护性。跨平台支持允许创建各种应用程序 - 桌面应用程序、软件即服务，甚至移动应用程序。 Node.js 非常适合数据密集型和实时应用程序，因为它使用事件驱动、非阻塞 I/O 模型，使其轻量级且高效。拥有如此庞大的社区，可以使用大量 Node.js 包来简化和促进开发。

与浏览器的区别 ​

浏览器和 Node.js 都使用 JavaScript 作为它们的编程语言。 构建在浏览器中运行的应用与构建 Node.js 应用完全不同。 尽管它始终是 JavaScript，但存在一些使体验截然不同的关键差异。

从广泛使用 JavaScript 的前端开发者的角度来看，Node.js 应用带来了巨大的优势： 用一种语言对一切（前端和后端）进行编程的舒适度。

变化的是生态系统

  • API 访问： 在浏览器中，主要与 DOM（文档对象模型）和其他 Web 平台 API（如 Cookie）进行交互。而在 Node.js 中，这些浏览器特定的 API 是不可用的，取而代之的是一些服务器端特定的 API。
  • 文件系统访问： 在 Node.js 中，可以通过其模块系统访问文件系统，从而实现对文件的读写操作。而在浏览器中，没有直接的文件系统访问功能。
  • 环境控制： 在 Node.js 中，可以控制运行环境，因为可以选择特定的 Node.js 版本来运行应用程序。相比之下，在浏览器环境中，无法控制访问者使用的浏览器版本。
  • JavaScript 版本支持： 在 Node.js 中，可以使用所有现代 ES2015+ JavaScript 功能，因为可以选择支持这些功能的 Node.js 版本。而在浏览器中，由于不同浏览器的升级速度不同，有时只能使用较旧的 JavaScript / ECMAScript 版本（可以使用Babel在交付浏览器运行前进行代码转换）。
  • 模块系统： Node.js 同时支持 CommonJS 和 ES 模块系统，意味着既可以使用require也可以使用import（从 Node.js v12 起）。而在浏览器中，正在实现 ES 模块标准，因此只能使用 import 语法。

运行Node.js代码 ​

运行 Node.js 程序的常用方法是运行全局可用的node命令（安装 Node.js 后）并传递要执行的文件的名称。

模块 ​

CommonJS VS ES Modules ​

说明

  1. CommonJS：

    • CommonJS是Node.js最早采用的模块系统，也是在许多Node.js应用和库中广泛使用的模块规范。
    • CommonJS使用require函数导入模块，使用module.exports或exports导出模块。
    • 它是一种同步模块加载的机制，即在加载模块时会阻塞执行线程，直到模块加载完成。
    • CommonJS适用于服务器端开发，因为在服务器端，模块加载时可以从本地文件系统同步读取。

  2. ES Module：

    • ES Module是JavaScript的官方模块系统，从ECMAScript 2015（ES6）开始引入，并逐渐成为了现代JavaScript应用的标准。
    • ES Module使用import语句导入模块，使用export关键字导出模块。
    • 它是一种异步模块加载的机制，即在加载模块时不会阻塞执行线程，而是使用异步加载方式，可以并行加载多个模块。
    • ES Module适用于浏览器端开发，并且可以通过工具（如Webpack、Rollup等）进行打包和转换，以支持在Node.js中使用。

确定模块类型

  • ESM:

    • 显式标记
      • .mjs 文件扩展名
      • 值为 "module" 的 package.json "type" 字段
      • 值为 "module" 的 --input-type 标志
      • 值为 "module" 的 --experimental-default-type 标志
    • 当代码没有显示标志说明时，如果代码中包含ES6import、export或import.meta语句，则默认为 ESM

  • CommonJS:

    • 显式标记
      • .cjs 文件扩展名
      • 值为 "commonjs" 的 package.json "type" 字段
      • 值为 "commonjs" 的 --input-type 标志
      • 值为 "commonjs" 的 --experimental-default-type 标志
    • 不存package.json文件或者package.json文件中没有"type"字段（无论--experimental-default-type标志的值是什么）
    • 未指定--input-type、字符串作为参数传递给--eval或通过STDIN通过管道传递给node

主要区别

  • 语法：CommonJS使用require和module.exports，而ESM使用import和export。
  • 加载方式：CommonJS采用同步加载，阻塞执行线程，ESM采用异步加载，不阻塞执行线程。
  • 浏览器支持：CommonJS在浏览器中不原生支持，需要通过工具转换成可执行的代码，而ESM在现代浏览器中得到原生支持。
  • 动态导入：ES Module支持动态导入，可以在运行时根据条件导入模块，而CommonJS只能在模块顶层进行静态导入。
  • 命名空间：ES Module会为每个模块创建单独的作用域，不同模块中的变量不会互相影响，而CommonJS模块共享同一个作用域。

根据以上语法和规则，可以自定义创建一个Node.js的模块

javascript
// 1. CommonJS

// 1.1 使用require函数导入模块

const fs = require("fs");

// 1.2 使用module.exports或exports导出模块

module.exports = {
  name: "CommonJS",
  age: 10,
};

// 2. ES Module

// 2.1 使用import语句导入模块

import fs from "fs";

// 2.2 使用export关键字导出模块

export default {
  name: "ES Module",
  age: 10,
};

global 全局命名空间对象 ​

文档位置

在浏览器中，顶层作用域传统上是全局作用域。这意味着var something将定义一个新的全局变量，ECMAScript 模块除外。在 Node.js 中，这是不同的。顶层作用域不是全局作用域；Node.js 模块内的var something对于该模块而言是本地的，无论它是 CommonJS 模块 还是 ECMAScript 模块。

官方文档指示为旧版的，使用globalThis代替

「globalThis MDN」 globalThis 提供了一个标准的方式来获取不同环境下的全局 this 对象（也就是全局对象自身）。不像 window 或者 self 这些属性，它确保可以在有无窗口的各种环境下正常工作。所以，你可以安心的使用 globalThis，不必担心它的运行环境。为便于记忆，你只需要记住，全局作用域中的 this 就是 globalThis。

异常处理 ​

uncaughtException事件 ​

中文文档位置

当未捕获的 JavaScript 异常一直冒泡回到事件循环时，则会触发 'uncaughtException' 事件。默认情况下，Node.js 通过将堆栈跟踪打印到 stderr 并以代码 1 退出，覆盖任何先前设置的 process.exitCode 来处理此类异常。为事件添加回调函数会覆盖此默认行为，回调函数接收以下两个参数：

  • err: Error 未捕获的异常。（使用Error对象，关于Error对象「点击这里」）
  • origin: String 指示异常来自未处理的rejection还是源自同步错误。值为unhandledRejection或uncaughtException。
javascript
const process = require("node:process");

process.on("uncaughtException", (err, origin) => {
  fs.writeSync(
    process.stderr.fd,
    `Caught exception: ${err}\n` + `Exception origin: ${origin}`
  );
});

setTimeout(() => {
  console.log("This will still run.");
}, 500);

// 故意引起异常，但不捕获它。
nonexistentFunc();

console.log("This will not run.");

注意:

  • uncaughtException是异常处理的最后手段，不能依赖它进行应用的异常处理，影响应用运行的异常应在代码中进行捕获和处理。
  • 此事件回调函数中的异常不会被捕获。
  • 尝试在未捕获的异常后正常恢复可能会影响系统寿命。正确用法是在关闭进程之前对分配的资源进行同步清理，在它之后恢复正常操作是不安全的。

uncaughtExceptionMonitor事件

  • uncaughtExceptionMonitor事件在uncaughtException事件触发或通过process.setUncaughtExceptionCaptureCallback()安装的钩子被调用之前触发。
  • 一旦触发 'uncaughtException' 事件，则安装 'uncaughtExceptionMonitor' 监听器不会更改行为。如果没有安装 'uncaughtException' 监听器，则进程仍然会崩溃。
  • 参数与uncaughtException事件相同
javascript
const process = require("node:process");

process.on("uncaughtExceptionMonitor", (err, origin) => {
  MyMonitoringTool.logSync(err, origin);
});

// 故意引起异常，但不捕获它。
nonexistentFunc();
// 依然会崩溃

错误类型 ​

中文文档位置 大多数内容和JavaScript的Error对象相同，关于Error对象「点击这里」

  • 标准的JavaScript错误
    • JavaScript-Error
  • 底层操作系统约束触发的系统错误。
  • 由应用代码触发的用户指定的错误。
  • AssertionErrors（断言错误）是一种特殊的错误。当Node.js检测到不应该发生的异常逻辑违规时会触发此错误。通常由node:assert模块引发。

Node.js引发的所有JavaScript错误和系统错误都继承自标准JavaScript Error类，或者是标准JavaScript Error类的实例，并且保证至少提供该类上可用的属性。

观察者模式是一种设计模式，用于实现对象之间的松耦合通信。在观察者模式中，有一个被观察的主题（Subject），以及一组观察者（Observers）订阅该主题。当主题的状态发生变化时，它会通知所有的观察者，观察者根据接收到的通知做出相应的响应。 事件触发器（EventEmitter）在 Node.js 中就是用于实现观察者模式的一种机制。它允许开发者定义事件和事件处理程序，当事件发生时，触发器会通知所有注册的事件处理程序。这里的事件处理程序就是观察者，它们对事件做出响应。

javascript
const EventEmitter = require("events");

// 创建一个新的事件发射器实例
const emitter = new EventEmitter();

// 定义事件处理程序
emitter.on("myEvent", () => {
  console.log("Event occurred!");
});

// 触发事件
emitter.emit("myEvent");

fs模块 ​

fs 模块提供了一个 API，用于以一种类似标准 POSIX 函数的方式与文件系统进行交互。

fs 和 fs/promise 用法相同，只是 fs/promise 使用 Promise API 而不是回调函数。这里只列举fs的几个常用用法，更多的方法使用可以参考文档，或者使用Dash快速查看

  • fs.readFile(path[, options], callback)：异步读取文件
  • fs.readFileSync(path[, options])：同步读取文件
  • fs.writeFile(file, data[, options], callback)：异步写入文件
  • fs.writeFileSync(file, data[, options])：同步写入文件
  • fs.appendFile(path, data[, options], callback)：异步追加文件
  • fs.chmod(path, mode, callback)：异步更改文件权限
  • fs.chmodSync(path, mode)：同步更改文件权限
  • fs.chown(path, uid, gid, callback)：异步更改文件所有者
  • fs.chownSync(path, uid, gid)：同步更改文件所有者

path模块 ​

path 模块提供了一些用于处理文件路径的工具。列举几个常用用法，更多的方法使用可以参考文档，或者使用Dash快速查看

  • path.join([...paths])：使用平台特定的分隔符作为定界符将所有给定的 path 片段连接在一起，然后规范化生成的路径。
  • path.resolve([...paths])：将路径或路径片段的序列解析为绝对路径。

关闭进程 ​

  • process.exit([code])：结束进程，code默认为0，表示成功，其他值表示失败
  • process.on('SIGINT', callback)：监听SIGINT信号，当用户按下ctrl+c时触发或者其他需要退出的事件，可以在回调函数中处理一些清理工作，然后调用process.exit()结束进程
javascript
// 1. process.exit([code])
// 1.1 退出成功
process.exit(0);
// 1.2 退出失败
process.exit(1);
// 2. process.on('SIGINT', callback)
process.on("SIGINT", () => {
  console.log("Received SIGINT. Press Control-D to exit.");
  // 关闭文件、数据库链接等
  process.exit(0);
});

接收输入 ​

  • process.stdin
    • process.stdin.on('data', callback)：监听用户输入，当用户输入时触发回调函数，回调函数接收一个参数data，表示用户输入的内容
    • process.stdin.on('end', callback)：监听用户输入结束，当用户输入结束时触发回调函数
    • process.stdin.pause()：暂停监听用户输入
javascript
// 1. process.stdin.on('data', callback)
process.stdin.on("data", (data) => {
  console.log(`User input: ${data}`);
});
// 2. process.stdin.on('end', callback)
process.stdin.on("end", () => {
  console.log("End of user input");
});
// 3. process.stdin.pause()
process.stdin.on("data", (data) => {

应用的可用性 ​

Node.js 的服务端程序在发生未捕获的错误时，默认情况下会导致进程崩溃。这是因为 Node.js 的设计哲学之一是快速失败（fail fast），即在发生错误时迅速停止程序的执行，以防止进一步的问题。

当未捕获的异常发生时，Node.js 会抛出一个未捕获异常错误（Uncaught Exception Error），并将其打印到控制台。随后，Node.js 进程会立即退出。

这种默认行为对于开发和调试过程非常有用，因为它可以帮助我们快速发现和解决问题。然而，在生产环境中，服务端程序的崩溃可能会导致服务不可用，因此我们通常希望能够更好地处理错误，而不是直接崩溃。

可以使用进程管理工具监控你的 Node.js 服务，自动重启进程以保持应用的可用性。

  • pm2

    • PM2 是 Node.js 应用程序的生产流程管理器，具有内置负载均衡器。它允许您使应用程序永远保持活动状态，无需停机即可重新加载它们，并简化常见的系统管理任务。
    • 文档

  • forever

    • Forever 是一个 Node.js 包，用于确保给定的脚本即使在服务器崩溃/停止时也能持续运行（即永远）。用于生产环境的 CLI 工具，用于管理 Node 应用程序及其进程。
    • 文档

const cluster = require("node:cluster");

什么是死锁?

死锁是指两个或两个以上的进程在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力作用，它们都将无法推进下去。这种现象我们称之为死锁。

javascript
const {
  Worker,
  isMainThread,
  parentPort,
  workerData,
} = require("node:worker_threads");

if (isMainThread) {
  module.exports = function parseJSAsync(script) {
    return new Promise((resolve, reject) => {
      const worker = new Worker(__filename, {
        workerData: script,
      });
      worker.on("message", resolve);
      worker.on("error", reject);
      worker.on("exit", (code) => {
        if (code !== 0)
          reject(new Error(`Worker stopped with exit code ${code}`));
      });
    });
  };
} else {
  const { parse } = require("some-js-parsing-library");
  const script = workerData;
  parentPort.postMessage(parse(script));
}

调试 ​

为什么不使用console.log()调试?

使用console.log调试代码通常会陷入“停止应用程序并添加console.log，然后再次启动应用程序”操作的无限循环。除了减慢应用程序的开发速度之外，它还会使编写变得肮脏并创建不必要的代码。最后，尝试注销变量以及其他潜在日志操作的噪音可能会使调试过程在尝试查找正在调试的值时变得困难。

  • 内存泄漏
    • 当 Node.js 应用程序的 CPU 和内存使用量随着时间的推移无明显原因增加时，就会导致内存泄漏。简单来说，Node.js 内存泄漏是堆上的一个孤立内存块，它不再被您的应用程序使用，因为它尚未被垃圾收集器释放。这是一块无用的内存。这些块可能会随着时间的推移而增长，并导致您的应用程序因内存不足而崩溃。
  • node --inspect: 通过使用--inspect标志启动 Node.js 应用程序，可以在 Chrome 开发者工具中调试 Node.js 应用程序。这是一个非常强大的工具，可以帮助您在开发过程中快速识别和解决问题。
  • 使用APM
    • 在Node.js中，APM代表应用性能管理（Application Performance Monitoring）。APM是一种用于监测和优化应用程序性能的方法和工具集合。
    • APM工具提供了对应用程序运行时性能的实时监控和分析。它们能够追踪并收集应用程序的关键指标，如请求响应时间、内存使用、CPU利用率、数据库查询等，以帮助开发者了解应用程序的性能状况和瓶颈。
    • Node.js中的APM工具通常提供以下功能：
      • 事务追踪：APM工具能够追踪应用程序处理请求的完整路径，包括内部函数调用和外部服务调用。这样可以帮助开发者识别慢速请求、高延迟操作或低效的代码路径。
      • 错误追踪：APM工具可以自动捕获和记录应用程序中发生的错误和异常。这有助于开发者快速定位和修复bug，并提供更好的应用程序质量和稳定性。
      • 性能分析：APM工具提供了性能分析功能，可识别应用程序的性能瓶颈、高负载请求和资源消耗过高的操作。通过分析性能数据，开发者可以优化关键代码路径和资源使用，提高应用程序的响应性能和可伸缩性。
      • 实时监控和警报：APM工具可以实时监控应用程序的性能指标，并根据预设的阈值触发警报。这样可以帮助开发者及时发现和解决性能问题，确保应用程序的可用性和稳定性。
    • 常见的Node.js APM工具包括New Relic、Datadog、AppDynamics、Elastic APM等。这些工具可以通过集成到应用程序中或使用代理进行监控，提供了丰富的性能数据和分析功能，帮助开发者不断改进和优化Node.js应用程序的性能。

  • console：在控制台中记录信息。
  • assert：提供一组断言测试。
  • process：提供有关当前进程的信息和对当前进程的控制。
  • cluster：能够创建同时运行并共享同一服务器端口的子进程。
  • perf_hooks：提供用于性能测量的API crypto：处理 OpenSSL 加密函数。
  • Buffer：提供 API 来处理二进制数据流。
  • DNS：启用名称解析。
  • events：用于处理现有事件和创建自定义事件。
  • child_processes：提供生成子进程的能力。
  • REPL：提供读取-评估-打印循环 (REPL) 实现，它既可以作为独立程序使用，也可以包含在其他应用程序中。
  • readline：提供一个接口，用于从可读流中一次一行读取数据。
  • util：支持 Node.js 内部 API 的需求。
  • querystring：提供用于解析和格式化 URL 查询字符串的实用程序。
  • string_decoder：提供用于将 Buffer 对象解码为字符串的 API。
  • tls：提供传输层安全（TLS）和安全套接字层（SSL）协议的实现。
